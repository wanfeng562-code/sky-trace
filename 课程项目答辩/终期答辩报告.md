## 一、核心业务逻辑

### 1.1 项目定位

Sky-Trace 是一个**面向桌面/Web 端的全球航班实时跟踪可视化平台**，以 FlightAware 为参照，整合多源公开航班数据与气象数据，为用户提供：

- 全球在飞航班的**实时位置监控**（~6,000 架次同时在线）；
- 对任意航班的**详情查阅与历史轨迹**查看；
- 全局航班**统计分析**；
- 指定时间范围内的**历史回放**。

### 1.2 业务流程总览

```
外部数据源层
  ├── OpenSky Network（OAuth2）  ──→  实时位置/速度/高度/航向，每 90 秒一次
  ├── OpenWeatherMap             ──→  枢纽机场天气 + GRD 格点天气，每 5 分钟
  ├── AirLabs（商业层）           ──→  机型/航班号/起降机场/时刻表，每 24 小时
  └── FlightRadar24（CF Worker） ──→  补充机队数据，后台独立循环

           ↓  统一采集管道（unified_pipeline）

后端服务层
  ├── 内存快照（FlightStore）   ──→  毫秒级查询当前机队
  ├── SQLite 持久化             ──→  轨迹历史 / 回放快照 / 机场目录 / 地名缓存
  └── WebSocket 广播            ──→  事件驱动推送，≤ 2 秒延迟

           ↓  REST + WebSocket（127.0.0.1:8000）

前端可视化层
  ├── 实时地图（MapView）       ──→  6,000+ 飞机图标实时渲染
  ├── 数据统计（StatsView）     ──→  ECharts 图表
  └── 历史回放（PlaybackView）  ──→  时间轴控制 + 尾迹动画
```

### 1.3 三层数据融合逻辑

系统将多源异构数据统一为**三层**，分别以不同频率采集：

| 层级   | 数据源          | 内容                          | 频率       |
| ------ | --------------- | ----------------------------- | ---------- |
| 实时层 | OpenSky Network | 位置/速度/高度/航向/ICAO24    | 90 秒/次   |
| 环境层 | OpenWeatherMap  | 枢纽天气 + GRD 网格天气 + AQI | 5 分钟/次  |
| 商业层 | AirLabs / FR24  | 机型/航班号/起降机场/时刻表   | 24 小时/次 |

三层数据在 `FlightStore` 中以 ICAO24 为键合并，向前端下发统一的 `FlightBrief` / `FlightDetail` 数据模型。商业层数据不影响实时层采集节奏，FR24 作为独立后台协程运行。

### 1.4 数据库核心设计

| 表名                   | 主要字段                                                              | 说明                                 |
| ---------------------- | --------------------------------------------------------------------- | ------------------------------------ |
| `flights`              | flight_id / callsign / icao24 / lat / lon / alt / speed               | 当前机队快照                         |
| `tracks`               | flight_id / ts / lat / lon / alt / speed（唯一索引）                  | 历史轨迹（60 秒节流落库）            |
| `flight_details_extra` | flight_id / aircraft_type / departure / arrival / dep_time / arr_time | 商业层 enrichment                    |
| `flight_snapshots`     | snapshot_time / data JSON                                             | 全机队快照（300 秒/次，24 小时 TTL） |
| `airports`             | iata / name / city / country / lat / lon                              | 8800+ 全球机场目录                   |
| `place_names`          | cache_key / name_zh / name_en / updated_at                            | 前后端双向地名缓存                   |

## 三、技术难点及解决方法

### 3.1 【P0 阻断】SQLite `tracks` 表膨胀导致服务无法启动

**现象**：`python run.py` 报 `sqlite3.IntegrityError: UNIQUE constraint failed: tracks.flight_id, tracks.ts`，服务启动即崩溃。

**根因分析**：
1. 早期代码每个 OpenSky tick 对同一 `(flight_id, ts)` 重复 `INSERT`，实测 `tracks` 表膨胀至约 518 万行 / 2.5 GB；
2. `init_db()` 的 `executescript(_DDL)` 中直接 `CREATE UNIQUE INDEX`，遇到重复行必然失败，且整段脚本全部回滚，服务无法启动。

**解决方案**（`server/app/services/db.py`）：

```python
async def _migrate_tracks_unique_index(conn):
    # 1. 检查索引是否已存在 → 跳过
    # 2. 尝试直接 CREATE UNIQUE INDEX → 无重复则立即成功
    # 3. 有重复 → ROW_NUMBER() OVER (PARTITION BY flight_id, ts)
    #    保留 MIN(id) 行，临时表重建，再建唯一索引
```

写入侧增加双重防护：
- `INSERT OR IGNORE INTO tracks ...`（数据库层去重）
- `_should_persist_track_to_db()`：同一航班最小落库间隔 60 秒（节流层去重）

**效果**：存量大库首次迁移约数分钟，之后每次启动约 0.1 秒；tracks 日增量从数十万行降至数千行。

### 3.2 高频 setData 导致地图渲染卡顿

**现象**：实时模式下每 90 秒 WebSocket 推送一次，前端对整个 GeoJSON Source 调用 `setData()`，同时选中/收藏/悬停三类高亮也触发 `setData()`，页面帧率明显下降。

**根因**：MapLibre `setData()` 会在 WebGL 层重新上传 GPU 纹理，每次调用开销约 5–30ms；多路高频触发形成"帧堆积"。

**解决方案**：

**① feature-state 替代高亮的 setData**（`client/src/utils/flightFeatureState.ts`）：
```typescript
// 仅更新单个要素的渲染状态，不重传整个 Source
map.setFeatureState({ source: 'flights', id: flightId }, { selected: true });
// 图层 paint 使用表达式驱动颜色/大小
"icon-size": ["case", ["feature-state", "selected"], 1.5, 1.0]
```

**② rAF 批量调度**（`client/src/utils/mapUpdateScheduler.ts`）：
```typescript
// 同一帧内多次 setData 合并为一次
scheduleMapUpdate(() => {
  flightsSource.setData(newGeoJson);
  applyFilters();
});
```

**效果**：选中/标记/悬停操作不再触发全量 Source 重传；高频操作帧率从约 18fps 提升至稳定 50fps+。

---

### 3.3 FR24 IP 封禁导致商业数据断源

**现象**：FR24 接口频繁返回 403 / 429，且封禁 IP 后整个采集管道受阻。

**根因**：FR24 数据抓取与 OpenSky 主采集在同一协程中串行执行，FR24 超时阻塞 OpenSky tick。

**解决方案**：

1. **FR24 独立后台协程**（`_fr24_background_loop`）：与 OpenSky 主 tick 完全解耦，FR24 被封不影响实时数据；
2. **Cloudflare Worker 代理**：前缀转发 `https://sky-trace-562.sweet-sky-4167.workers.dev/?url=`，规避服务器 IP 的直连封禁；
3. **FR24 数据独立命名空间**：通过 `FlightStore.apply_fr24_snapshot()` 写入 `fr24-*` 前缀的条目，与主力 `icao24-*` 机队完全隔离。

### 3.4 多维筛选下 MapLibre 过滤器的边界问题

**现象**：取消最后一个收藏标记后，收藏高亮圈仍然渲染，无法消失。

**根因**：使用 `["in", "flight_id", ""]`（空字符串）作为空集合过滤器时，MapLibre 会错误匹配 `flight_id` 字段为空字符串的要素。

**解决方案**（`client/src/utils/mapFilters.ts`）：
```typescript
export function markedFlightsFilter(ids: readonly string[]): FilterSpecification {
  const list = ids.filter(id => typeof id === 'string' && id.length > 0);
  if (!list.length) return false;  // false = 隐藏所有要素，而非误匹配
  return ["in", ["get", "flight_id"], ["literal", list]];
}
```
关键：空集合时返回布尔 `false`，不是空字符串 `""`。

### 3.5 API 额度管控与多源协作

**现象**：OpenSky 每日 4,000 积分，OpenWeather / AirLabs 免费层各约 1,000 次/日，稍不注意即超额。

**解决方案**：
- **分档配置**：`APP_PROFILE=development` 时 OpenSky 固定 90 秒/次（全天 ~960 次 × 4 积分 = 3,840 积分，留有余量）；
- **环境网格上限**：`environment_grid_max_cells = 40`，每轮 GRD 天气最多 40 个格点；
- **AirLabs 24 小时缓存**：商业层数据每日仅刷新一次；
- **API 软截断**：`daily_call_budget` 超限后停止采集，防止意外超额产生费用；
- **前端持久化缓存**：`persistentCache.ts` 对机场数据缓存 30 分钟、行政边界缓存 7 天、地名缓存 90 天，大幅减少重复请求。

### 3.7 DataV 地图 API 跨域 403——行政边界数据的多级降级策略

**现象**：本地开发时中国省级行政边界高亮正常；部署到 `sky-trace.pages.dev` 后，选中省份只显示矩形框而非真实边界。

**根因分析**：  
前端通过 `geo.datav.aliyun.com/areas_v3/bound/{adcode}_full.json` 获取省级边界多边形。该 API 归属阿里云，对**海外域名**（Cloudflare Pages）的跨域请求返回 `403 Forbidden`，但允许 `localhost` 访问，导致本地正常、线上异常的典型跨源隔离问题。

**解决方案**（`client/src/utils/regionBoundaries.ts`，commit `b5ca87c`）：

构建三级降级链：

```
DataV API（高精度，国内域名/localhost 可用）
    ↓ 403 / 网络失败
bundled cn-admin1.json（Natural Earth 数据，已打包进 public/geo/，无 CORS 限制）
    ↓ 未收录（HK / TW / MO）
bbox 矩形回退（不写入缓存，下次仍重试）
```

```typescript
// client/src/utils/regionBoundaries.ts
geom = await fetchDatavBoundary(adcode);
if (!geom && !specific) {
  // DataV 被海外域名阻断 → 回退到本地打包 GeoJSON
  if (adcode === "100000") {
    geom = await fetchCountryAdmin1Outline("CN");        // 整体国境线
  } else {
    geom = await fetchAdmin1ByRegionCode("CN", regionCode ?? "", bbox); // 省级
  }
}
if (!geom) fetchFailed = true;   // 不缓存，允许重试
```

**设计要点**：
- `cn-admin1.json` 已静态打包（32 个大陆省份，Natural Earth 精度），无额外网络请求；
- 香港、台湾、澳门未收录于该文件，回退为 bbox——面积极小，误差视觉可忽略；
- `fetchFailed` 标志位确保降级结果不写入 7 天 localStorage 缓存，DataV 恢复后自动升级至高精度。

**效果**：线上 `sky-trace.pages.dev` 选中省份后高亮真实边界，与本地开发效果一致。

## 四、亮点与优化方法

### 4.1 零成本高质量地图底图

**亮点**：默认采用 **OpenFreeMap liberty**（矢量瓦片，无需 API Key，无访问限制）+ **AWS Terrarium DEM hillshade**（地形晕渲，公开 CDN），叠加双语地名（优先简体中文），达到接近 MapTiler 付费底图的视觉效果，且无任何密钥管理成本。

同时提供底图一键切换面板（OpenFreeMap / MapTiler / Stadia / Satellite），运行时无缝切换，不重置相机位置。

### 4.2 统一视觉主题——琥珀金飞机体系

**亮点**：全图所有飞机图标统一使用琥珀金 `#FFBF00`（`AIRCRAFT_AMBER`），配合深色底图形成高对比度，同时与枢纽机场圆点（绿/蓝/紫三档）形成两套视觉语言，一眼区分"动态目标"与"静态基础设施"。

视觉 Token 集中管理于 `client/src/config/mapVisualTheme.ts`，便于全局一致修改。

### 4.3 枢纽机场分级与缩放感知

**亮点**：全球 98 个枢纽机场按当前快照中的起降航班数实时划分 Tier 1/2/3（绿/蓝/紫），并随地图缩放级别动态截断显示数量（zoom 3 时仅显示 16 个最繁忙枢纽，zoom 8+ 显示全部），模拟 FlightAware 的"远少近多"显示逻辑。

### 4.4 四级行政区域联动筛选

**亮点**：`FlightFilterPanel` 实现了**国家 → 省/州 → 城市 → 区县**四级联动下拉筛选，结合高度范围、速度范围、飞行类型、机型类别共计 8 维筛选条件，实时更新地图显示。底层数据由 `geoNested.ts` 嵌套树 + `geoHierarchy.ts` 层级映射支撑。

### 4.5 地名双向持久化缓存

**亮点**：`placeNameSync.ts` 实现前后端**双向地名同步**：
- 前端翻译后的中文地名经 800ms debounce 自动写入后端 `place_names` 表；
- 下次访问时直接从服务端加载，无需重复翻译；
- `persistentCache.ts` 在前端 localStorage 额外缓存 90 天，断网时仍可使用已有翻译。

### 4.6 历史回放双模式

**亮点**：回放页面支持两种模式：
- **全局模式**：回放指定时间范围内所有航班的快照（`buildPlaybackCatalog()` 合并机队），时间轴滑动展示历史全貌；
- **单航班模式**：指定 `flight_id` 回放精细轨迹，`playbackTrail.ts` 仅追加新轨迹点（增量更新），不全量重建 LineString，渲染效率高。

### 4.7 前端组件深度拆分

### 5.2 演示路线建议

**路线一：实时监控（约 3 分钟）**
1. 打开主页 → 地图实时渲染全球航班，右上角显示实时架次数；
2. 在航班列表搜索 `CCA`（国航），观察实时过滤；
3. 点击地图上任意飞机图标 → 右侧面板展示详情（速度/高度/起降机场/天气）；
4. 点击详情中"查看轨迹" → 地图绘制该航班历史飞行路径；
5. 演示底图切换（OpenFreeMap → Satellite）+ 图层开关（开启气象格点）。

**路线二：筛选与分析（约 2 分钟）**
1. 打开高级筛选面板 → 设置高度范围 25000ft–45000ft，观察地图仅显示高空航班；
2. 再加上机型筛选 `A320`，观察进一步过滤；
3. 切换到"数据统计"页 → 展示飞行状态分布、高度/速度分布图表、前 20 航司排行。

**路线三：历史回放（约 2 分钟）**
1. 切换到"历史回放"页；
2. 选择时间范围（如过去 6 小时）→ 点击播放；
3. 拖动时间轴，观察全球机队历史位置变化；
4. 在回放目录中选择单架航班 → 切换单航班模式，观察尾迹增量动画。

**路线四：枢纽机场（约 1 分钟）**
1. 主地图页右侧面板切换到"枢纽"Tab → 枢纽列表按在飞航班数排序；
2. 点击"北京首都（PEK）"旁的定位按钮 → 地图飞行至北京；
3. 展开时刻表，查看当日进出港计划。
