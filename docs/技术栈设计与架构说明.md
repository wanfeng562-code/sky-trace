# Sky-Trace 技术栈设计与架构说明

## 1. 文档目标
本文档用于明确 Sky-Trace 的技术选型、模块边界与通信方式，确保后端与前端在同一技术基线上协作开发，并满足课程项目的可交付性与可演示性。

## 2. 总体架构
Sky-Trace 采用 **前后端分离** 架构：Vue3 SPA 前端 + Python FastAPI 后端，桌面壳（PySide6）作为可选次级入口。

- **可视化界面（主入口）**：Vue3 SPA 运行于浏览器或 Cloudflare Pages，提供地图、航班列表、图表分析与交互页面
- **数据服务**：Python FastAPI 负责数据采集、清洗、缓存、推送和业务规则处理
- **桌面壳（可选）**：PySide6 提供桌面应用窗口，通过 Qt WebEngine 嵌入前端构建产物；目前已降优先级，日常开发以浏览器直接访问前端为主

## 3. 版本基线与核心技术栈
为保证团队环境一致，版本基线统一如下。

### 3.1 运行时版本
- Python：3.10 或更高，建议 3.11.x
- Node.js：20 LTS（建议 20.11+）
- npm：10+
- Qt for Python：PySide6 6.7+

### 3.2 后端技术（Python）
- FastAPI：对外提供 HTTP 接口与 WebSocket 实时推送
- Uvicorn：ASGI 运行服务
- aiohttp：异步调用航班与天气外部接口
- asyncio 后台任务：统一采集实时层/环境层/商业层数据
- Pydantic：请求与响应数据模型校验
- SQLAlchemy（可选）+ SQLite：本地缓存与历史回放

### 3.3 前端技术（Vue）
- Vue3 + TypeScript + Vite：前端主框架与构建
- Vue Router：多页面路由
- Pinia：状态管理
- MapLibre GL JS 或 Leaflet：地图可视化与轨迹绘制
- ECharts：统计看板（航班数量、速度分布、高度分布等）

### 3.4 桌面壳技术（Qt，可选）
- PySide6：桌面应用入口（已降优先级，非主要开发方向）
- Qt WebEngine：嵌入 Vue 构建产物
- Qt WebChannel：Qt 与前端 JS 双向通信（如系统通知、设置、文件导出）

> **注意**：desktop/ 目录中的 PySide6 壳代码已归档维护。日常开发与演示均通过浏览器直接访问前端（`npm run dev` / Cloudflare Pages 部署），无需启动桌面壳。

## 4. 项目结构建议
建议采用如下目录划分，减少职责交叉并便于协作。

- client：Vue3 前端项目
- server：Python 后端服务
- desktop：PySide6 桌面壳
- docs：设计说明、协作规范与课程文档
- scripts：开发与发布脚本

## 5. 通信与数据流设计
Sky-Trace 的通信由三条通道组成。

### 5.1 前端与后端
- HTTP：用于筛选查询、统计接口、历史数据拉取
- WebSocket：用于实时航班增量推送
- 数据格式：统一 JSON，统一字段命名与时间标准
- 统一下发：前端优先调用后端聚合接口，不直接访问第三方 API

示例字段约定：
- 时间字段统一为 UTC ISO 8601（例如 2026-04-12T14:30:00Z）
- 经纬度使用浮点数（WGS84）
- 速度单位统一为 knot，高度单位统一为 feet

### 5.2 Qt 与前端
- Qt 启动后加载本地前端资源（dist）
- 通过 Qt WebChannel 暴露桥接对象给前端
- 典型场景：本地配置读取、日志导出、截图、窗口控制

### 5.3 Qt 与后端
- 本地进程方式：Qt 作为主进程拉起 Python 服务（推荐）
- Qt 负责检查后端健康状态与端口占用
- 若后端异常退出，Qt 提示重启并写入本地日志

## 6. 接口规范与契约管理
- 后端维护 OpenAPI 文档，作为前后端协作契约
- 前端仅依赖契约字段，不直接假设后端内部实现
- 新增字段优先向后兼容，删除字段必须走变更评审
- 接口版本建议前缀化，例如 /api/v1

## 7. 配置与环境管理
- 环境变量模板：`server/.env.development.example`、`server/.env.release.example`
- 运行档位：通过 `APP_PROFILE=development|release` 切换
- 严禁将 API Key、Token、数据库账号提交到仓库
- 前端通过 .env.development 管理后端地址
- 后端通过 .env 管理外部数据源与调度频率

开发档位实时间隔策略（OpenSky）：
- 全时段匀速：90 秒/次（约 960 次/日，消耗 3840 点/日，预算 4000 点/日）
- 无活跃/非活跃窗口区分（已废弃旧的"15s活跃+40s非活跃"双档设计，改为全天均速）

## 8. 日志与可观测性
- 后端日志分级：DEBUG、INFO、WARN、ERROR
- 关键行为必须有日志：抓取开始/结束、接口失败、数据异常
- 建议保存最近 N 小时抓取日志，便于演示与排障

## 9. 性能与扩展建议
- 地图点位采用分层渲染与可见范围加载
- 实时数据按区域与缩放级别进行采样
- 历史回放采用分页与时间窗口加载
- 后期可扩展 Redis 做热数据缓存

## 10. 里程碑建议
- M1：跑通最小链路（抓取 -> 后端 -> 前端地图展示）
- M2：增加筛选、轨迹、详情卡片与统计图
- M3：增加回放、异常航迹识别与桌面壳增强

## 11. 风险与应对
- 外部接口限流或不稳定：增加重试、降级与本地缓存
- 前后端字段不一致：以契约文档为单一事实来源
- 合并冲突频发：按模块边界分工，严格走 PR 审查

## 12. 最新落地快照（截至 2026-05-18）

### 12.1 后端

- **统一采集管道**：实时（OpenSky OAuth2 → mock 降级）、环境（DB 枢纽 + **GRD 5° 动态网格**，`environment_grid_max_cells` 限流）、商业（AirLabs 日批 + **FR24 仅后台循环合并**，不再每 tick 写库）。
- **SQLite 六表**：`flights`、`tracks`（唯一索引 `(flight_id,ts)` + `INSERT OR IGNORE` + 落库节流）、`flight_snapshots`、`flight_details_extra`、`airports`（~8800 IATA 目录）、`place_names`。
- **回放**：`persist_playback_snapshot` + `GET /api/v1/playback`；TTL 可配置。
- **扩展 API**：`GET /api/v1/weather-grid`、`GET/PUT /api/v1/places/names`、`GET /api/v1/airports/lookup`、瓦片代理 `/api/v1/tiles/*`。
- **WS**：`broadcast_manager` 并行发送；全量广播最小间隔 `ws_broadcast_min_interval_seconds`（默认 2s）。

### 12.2 前端

- **三页路由**：`/` MapView、`/stats` StatsView、`/playback` PlaybackView；壳层 `AppShell.vue` + **i18n**（zh-CN / en-US）。
- **地图组件化**：`components/map/*`（侧栏、控件、图层面板）、`components/flight/*`（列表、筛选、标记）。
- **性能与交互**：`mapFilters`（取消标记不误高亮）、`flightFeatureState`（选中/标记不用整源 setData）、`mapUpdateScheduler`、`persistentCache` + `placeNameSync`、`playbackTrail` 增量尾迹。
- **区域能力**：多国 admin1 GeoJSON、`regionBoundaries` 真实边界高亮、四级地区筛选（`flightFilters.ts`）。

### 12.3 部署

- 生产前端：**Cloudflare Pages**（https://sky-trace.pages.dev/）；后端云服务器 `0.0.0.0:8000`，CORS 需包含 Pages 域名。
- 详细修复清单见 [系统前后端全面修复与优化.md](./系统前后端全面修复与优化.md)。
