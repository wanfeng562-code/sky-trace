## 2. 系统架构速览

```
Vue3 前端 (MapView / PlaybackView / StatsView)
        │  REST + WebSocket
        ▼
FastAPI (unified_pipeline 统一采集)
        │
        ├── OpenSky OAuth2（实时位置）
        ├── OpenWeather（枢纽 + GRD 网格天气）
        ├── AirLabs / FR24 代理（商业 enrichment，后台合并）
        └── SQLite (flights / tracks / snapshots / airports / place_names)
```

**数据流要点**：

- 实时：`OpenSky` → 内存 `FlightStore` → WS 广播 + 节流写库；
- 环境：DB 枢纽 + 按活跃航班 5° 网格补盲（`GRD_{lat}_{lon}`）；
- 回放：定时全量快照 `flight_snapshots` + 可选 `tracks` 历史；
- 商业层：FR24 等仅在后台周期合并，避免每个 OpenSky tick 写库。

## 3. 后端修复与优化

### 3.1 SQLite：`tracks` 表膨胀与启动失败（P0）

**现象**  
`python run.py` 在 `init_db()` → `executescript(_DDL)` 报错：

```text
sqlite3.IntegrityError: UNIQUE constraint failed: tracks.flight_id, tracks.ts
```

**根因**

1. 优化前每个 OpenSky tick 对同一 `(flight_id, ts)` 重复 `INSERT`，库内可达数百万重复行（实测约 518 万行 / 2.5GB）。
2. 在 `executescript` 中对已有重复数据直接 `CREATE UNIQUE INDEX` 会失败，且整段脚本回滚导致**服务无法启动**。
3. 早期去重方案 `DELETE ... NOT IN (子查询)` 在大表上极慢，启动会长时间卡住。

**修复**（`server/app/services/db.py`）

1. 从 `_DDL` 中**移除**唯一索引创建，避免与 `executescript` 绑定失败。
2. 新增 `_migrate_tracks_unique_index()`：
   - 索引已存在 → 跳过；
   - 尝试直接建索引 → 成功则结束；
   - 失败则**表重建去重**（`ROW_NUMBER() OVER (PARTITION BY flight_id, ts ORDER BY id)`，保留 `MIN(id)` 行），再建 `idx_tracks_flight_ts_unique`。
3. 首次迁移大库约数分钟；之后 `init_db` 约 0.1s。

**写入侧防护**（`server/app/services/flight_store.py`）

- `INSERT OR IGNORE INTO tracks ...`
- `_should_persist_track_to_db()`：按 `tracks_persist_interval_seconds`（默认 60s）节流，同一航班不过频落库。

**配置**（`server/app/core/config.py`）

| 配置项 | 默认 | 说明 |
|--------|------|------|
| `tracks_persist_interval_seconds` | 60 | 同一航班轨迹落库最小间隔 |
| `tracks_ttl_hours` | 24 | 历史轨迹保留时长 |

### 3.2 采集管道：减少重复写库与 API 压力

**文件**：`server/app/services/unified_pipeline.py`

| 优化项 | 说明 |
|--------|------|
| FR24 后台合并 | 商业数据不再随每个 OpenSky tick 写 SQLite，仅在后台循环合并 |
| 批量 enrichment | `upsert_detail_extra_batch()` 单事务写入多条 `flight_details_extra` |
| 环境网格上限 | `environment_grid_max_cells`（默认 40）限制每轮 GRD 刷新数量 |
| 枢纽天气批量 | `_hub_weather_max_batch = 40`，避免单轮请求过多 |
| WS 广播节流 | `ws_broadcast_min_interval_seconds`（默认 2s），降低全量快照推送频率 |
| 回放快照 TTL | `persist_playback_snapshot()` 写入快照并清理过期 `tracks`/快照 |

**配置摘要**

| 配置项 | 默认 |
|--------|------|
| `playback_snapshot_interval_seconds` | 300 |
| `playback_ttl_hours` | 24 |
| `ws_broadcast_min_interval_seconds` | 2.0 |
| `environment_grid_max_cells` | 40 |

## 4. 前端修复与优化

### 4.1 地图标记取消后仍高亮（逻辑 Bug）

**现象**：取消标记后，高亮圈/筛选层仍显示该航班。

**根因**：MapLibre 的 `filter` 使用 `flight_id === ""` 或空数组时，可能误匹配要素。

**修复**（`client/src/utils/mapFilters.ts`）

```ts
export const HIDE_ALL_FEATURES_FILTER = false;

export function markedFlightsFilter(ids: readonly string[]): FilterSpecification {
  const list = ids.filter((id) => typeof id === "string" && id.length > 0);
  if (!list.length) return false;
  // ...
}
```

`MapView.vue` / `PlaybackView.vue` 统一引用，无标记时图层 filter 为 `false`，不渲染任何要素。

## 5. 环境与网格（GRD）说明

**GRD** = 5°×5° 动态天气网格点，ID 形如 `GRD_+25_-10`。

- **后端**：活跃航班所在格子若无枢纽覆盖，则拉取该格中心天气（受 `environment_grid_max_cells` 限制）。
- **前端**：`weatherGrid.ts` 区分枢纽与 `GRD_*`，避免把网格点当枢纽渲染；网格图层分 fill/line，支持航班变化后延迟刷新。

## 9. 故障排查速查

| 症状 | 可能原因 | 处理 |
|------|----------|------|
| 启动报 `UNIQUE constraint failed: tracks` | 旧库重复行 + DDL 建索引 | 已用迁移函数；确保使用最新 `db.py` 启动一次 |
| 启动卡在 `starting backend service` 很久 | 首次 tracks 去重迁移 | 等待完成；看日志 `rebuilding table` |
| 标记取消仍亮 | 旧前端 filter | 确认 `mapFilters.ts` 为 `false` 而非 `""` |
| 网格全灰 | `has_weather` 未命中 / API key | 检查 OpenWeather 配置与 `/api/v1/weather-grid` |
| 地名不同步 | `places` 路由未启用 | 确认 `main.py` 已 `include_router(places_router)` |
