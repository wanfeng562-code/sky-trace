## 4. 架构变化（改造前 vs 改造后）

### 4.1 数据来源
改造前：
- 机场元数据主要依赖后端代码常量，扩展成本高。

改造后：
- 机场元数据统一存储在 SQLite airports 表。
- 启动时支持空表自动种子导入。

### 4.2 采集策略
改造前：
- 环境采集点集合偏静态，覆盖弹性不足。

改造后：
- hub 由 DB 驱动动态调度。
- 增加 GRD 网格观测点（随活跃航班动态变化）。

### 4.3 API 聚合
改造前：
- /airports 不具备“DB 主数据 + 动态观测点统一视图”能力。

改造后：
- /airports 输出 DB 机场与动态 weather 点融合结果，并附 point_type。

### 4.4 前端渲染
改造前：
- 机场点位基本单层展示，语义分层不足。

改造后：
- MapView 将 hub 与 GRD 拆分图层并提供独立开关。
- 点击交互按点位类型分流，避免误触发业务动作。

## 5. 逐模块交接（全窗口关键实现）

## 5.1 后端数据库层
文件：server/app/services/db.py

关键变更：
- 新增 airports 表：iata_code、name、city、country、lat、lon、is_hub。
- 新增索引：is_hub、lat/lon。
- init_db 增量迁移：旧库自动补列。
- 新增 _seed_airports_if_empty：空表导入机场种子。
- 新增 list_airports(is_hub=None|True|False) 查询接口。

结果：
- 机场主数据从代码常量迁移到 DB。
- 运行时查询与扩展能力提升。

## 5.2 后端采集层
文件：server/app/services/unified_pipeline.py

关键变更：
- 删除/弃用静态 hub 依赖，改为 db_list_airports(is_hub=True) 输入。
- 新增 hub 调度状态：_hub_weather_next_due。
- hub 下次采集间隔随机 10 到 30 分钟。
- 每轮最多处理一定数量 due hub（批次控制）。
- OpenWeather 请求并发限制 + 随机抖动。
- hub 失败单次重试（延迟 5 秒）。
- 继续生成 GRD 网格点（5° 网格）并抓取天气。
- GRD 上限 _MAX_GRID=80。
- get_nearest_hub_weather 改为 DB 排序查询最近可用 hub。

结果：
- 采集点从“固定名单”变为“DB 主驱动 + 动态网格补盲”。

## 5.3 后端接口层
文件：server/app/api/routes/flights.py

关键变更：
- /airports 改为 DB 机场 + weather_cache 动态点聚合。
- 输出字段扩展：city、country、is_hub、point_type。
- point_type 语义：hub / airport / grid / weather。
- 对 GRD 点命名区分，便于前端与用户识别。

结果：
- 前端可基于统一接口做分层渲染。
- 点位来源可追踪，业务语义明确。
