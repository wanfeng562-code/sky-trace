## 一、需求说明书

### 1.1 项目背景与目标

随着全球航空运输业持续增长，实时掌握航班位置、状态和气象信息具有重要意义。本项目旨在开发一个**面向桌面端的航班信息跟踪可视化平台**，整合多源公开航班数据与气象数据，为用户提供全球实时航班追踪、历史轨迹回放与数据统计分析等功能。

### 1.2 用户角色

| 角色       | 描述                   | 核心需求                           |
| ---------- | ---------------------- | ---------------------------------- |
| 普通用户   | 对航班动态感兴趣的个人 | 实时查看航班位置、搜索特定航班详情 |
| 航空爱好者 | 深度关注航空数据的用户 | 历史轨迹分析、机型识别、统计报表   |
| 课程评审者 | 验收项目的教师         | 功能完整性、系统稳定性、界面美观度 |

## 二、详细设计

### 2.1 技术方案

#### 2.1.1 系统架构总览

Sky-Trace 采用**三层混合架构**，分离关注点：

```
┌──────────────────────────────────────────────┐
│            桌面壳层（PySide6 / Qt WebEngine）          │
│  进程管理 │ 窗口容器 │ WebChannel 本地桥接          │
└─────────────────┬────────────────────────────┘
                  │ 嵌入（localhost:5173 或 dist/）
┌─────────────────▼────────────────────────────┐
│         前端可视化层（Vue3 / TypeScript）           │
│  MapLibre地图 │ 航班列表 │ 详情卡片 │ 统计图表      │
│  Pinia状态管理 │ WebSocket客户端 │ HTTP API调用    │
└─────────────────┬────────────────────────────┘
                  │ HTTP REST + WebSocket（127.0.0.1:8000）
┌─────────────────▼────────────────────────────┐
│          后端数据服务层（FastAPI / Uvicorn）          │
│  ┌────────────┬──────────────┬─────────────┐  │
│  │  实时层     │   环境层       │   商业层      │  │
│  │ OpenSky    │ OpenWeather  │  AirLabs    │  │
│  │ 90s/次     │ 5min/次      │ 24h/次      │  │
│  └────────────┴──────────────┴─────────────┘  │
│  SQLite持久化 │ 内存快照缓存 │ BroadcastManager│  │
└──────────────────────────────────────────────┘
```

#### 2.1.2 核心技术选型说明

**后端：FastAPI + Uvicorn**  
选择 FastAPI 的原因：原生支持 async/await 全异步处理，与 aiohttp 外部调用无缝集成；自动生成 OpenAPI 文档；Pydantic 数据校验减少运行时错误。

**前端：Vue3 + MapLibre GL JS**  
MapLibre GL JS 是 Mapbox GL 的开源 fork，支持矢量瓦片渲染和 WebGL 硬件加速，可流畅渲染数千个动态图层。Vue3 的 Composition API 和 Pinia 状态管理保证了数据流的可预测性。

**数据库：SQLite + aiosqlite**  
单文件轻量级方案，满足课程项目的持久化需求（轨迹历史、快照回放）而无需额外部署数据库服务。WAL 模式支持并发读写。

**实时通信：WebSocket（事件驱动）**  
后端采集完成后由 BroadcastManager 主动推送，而非前端定时轮询，减少不必要的网络请求，降低端到端延迟。

#### 2.1.3 关键设计决策

**三层数据采集管道**  
将实时层（高频/低延迟）、环境层（中频）、商业层（低频）解耦，各层独立失败不影响其他层，实现优雅降级。

**OAuth2 Token 管理**  
`_get_opensky_auth()` 方法统一管理 Bearer Token 生命周期：自动缓存、提前30秒续期、降级链（OAuth2 → Basic Auth → 匿名），保证认证透明切换。

**Mock数据兜底**  
所有外部API均配置 `REALTIME_FALLBACK_TO_MOCK=true`，确保API不可用时演示不中断。

### 2.2 功能设计

#### 2.2.1 数据采集流程

```
启动 unified_pipeline.start()
    ↓
APScheduler 触发 _collect_realtime()
    ↓
    ├─ OpenSky /states/all → 解析 states[][] 数组
    │   ↓
    │   FlightBrief 标准化（icao24→flight_id, baro_altitude→altitude_ft）
    │   ↓
    │   FlightStore.update_snapshot() → SQLite upsert + 轨迹append
    │   ↓
    │   BroadcastManager.broadcast() → 所有WS客户端推送 snapshot 事件
    │
    ├─ _collect_environment() （独立调度，5分钟/次）
    │   ↓
    │   asyncio.gather(*[fetch_hub(hub) for hub in 20_HUBS])
    │   天气+AQI → _weather_cache[IATA] / _air_quality_cache[IATA]
    │
    └─ _collect_commercial() （独立调度，24小时/次）
        ↓
        AirLabs /flights → 按呼号匹配 → flight_details_extra 写入SQLite
```

#### 2.2.2 前端状态管理

```
Pinia flightStore
    ├─ flights: FlightBrief[]          ← WS snapshot 更新
    ├─ searchKeyword: string           ← 搜索框输入
    ├─ filterStatus: all|airborne|...  ← 筛选 Tab
    ├─ selectedFlightId: string|null   ← 当前选中
    ├─ flightDetail: FlightDetail|null ← HTTP 拉取详情
    ├─ wsOnline: boolean               ← WS连接状态
    └─ filteredFlights (computed)      ← 双过滤结果，地图+列表共用
```

#### 2.2.3 地图图层叠加顺序（z-order）

由低到高依次为：  
底图（矢量瓦片）→ hillshade（地形晕渲）→ 机场圆点 → 机场高亮圆点 → 轨迹线 → 选中光晕 → 飞机图标（最顶层）

### 2.4 数据库设计

**数据库类型：** SQLite（WAL模式），文件路径：`server/data/sky_trace.db`

#### 表：`flights`（航班最新位置快照）

| 字段        | 类型             | 说明                  |
| ----------- | ---------------- | --------------------- |
| flight_id   | TEXT PRIMARY KEY | ICAO24 应答机编码     |

| lat         | REAL NOT NULL    | 纬度（WGS84）         |

| speed_kts   | INTEGER          | 地速（节）            |
| altitude_ft | INTEGER          | 气压高度（英尺）      |
| updated_at  | TEXT NOT NULL    | UTC时间戳（ISO 8601） |

| 字段        | 类型                              | 说明                   |

| id          | INTEGER PRIMARY KEY AUTOINCREMENT | 自增主键               |
| flight_id   | TEXT NOT NULL                     | 外键→flights.flight_id |
| ts          | TEXT NOT NULL                     | 采集时间戳             |
| lat         | REAL NOT NULL                     | 纬度                   |
| lon         | REAL NOT NULL                     | 经度                   |
| altitude_ft | INTEGER                           | 高度（英尺）           |
| speed_kts   | INTEGER                           | 速度（节）             |

#### 表：`flight_snapshots`（整队快照，用于回放）

## 三、风险分析

### 3.1 技术风险

| 风险                    | 等级 | 描述                                                           | 缓解措施                                                        |
| ----------------------- | ---- | -------------------------------------------------------------- | --------------------------------------------------------------- |
| 外部API限流/不可用      | 高   | OpenSky 在高频请求下会返回 429，OpenWeather/AirLabs 有配额限制 | Mock数据降级；OAuth2提升额度至4000积分/日；soft cap 防超额      |
| 中国航班覆盖不足        | 高   | OpenSky 依赖ADS-B地面站覆盖，中国及俄罗斯等国接收站少          | 后续探查第二数据源（如FlightRadar24 API、航空公司官方数据）补充 |
| WebSocket长连接稳定性   | 中   | 网络波动导致断线；服务器重启                                   | 退避重连策略（2/4/8/15/30s）；心跳 ping 保活                    |
| OAuth2 Token 过期       | 中   | Token默认1800s，缓存不当导致401                                | 提前30秒续期；失败时立即重新获取                                |
| Qt WebEngine 渲染兼容性 | 低   | 特定 PySide6 版本的 WebEngine 可能与 MapLibre GL WebGL 不兼容  | 提供浏览器独立访问模式（127.0.0.1:5173）作为降级方案            |

### 3.2 成本风险

| 风险                | 等级 | 描述                                   | 缓解措施                                                                 |
| ------------------- | ---- | -------------------------------------- | ------------------------------------------------------------------------ |
| API配额耗尽         | 中   | 演示前额度不足，实时数据无法加载       | 设置每日调用上限1000次；UTC 00:00（北京时间08:00）重置；提前备好mock数据 |
| AirLabs免费层不稳定 | 低   | 免费层月调用约1000次，密集测试可能耗尽 | AirLabs仅每24小时采集一次（约30次/月），余量充足                         |

### 3.3 市场/项目风险

| 风险         | 等级 | 描述                                        | 缓解措施                                         |
| ------------ | ---- | ------------------------------------------- | ------------------------------------------------ |
| 功能范围蔓延 | 中   | 三周内不断扩展需求导致核心功能不稳定        | P0功能优先保障，P2功能按时间允许递进完成         |
| 团队协作冲突 | 低   | 前后端接口不同步导致联调卡顿                | 契约先行（API文档先冻结），使用Git功能分支隔离   |
| 演示环境依赖 | 中   | 现场演示需网络访问OpenSky，断网则无实时数据 | Mock降级保证演示不中断；提前本地录制演示视频备用 |

### 3.4 已解决的风险

| 风险                                 | 解决时间   | 解决方法                                           |
| ------------------------------------ | ---------- | -------------------------------------------------- |

### 10.4 部署步骤

```powershell
# 1. 克隆仓库
git clone https://gitee.com/[用户名]/Sky-Trace.git
cd Sky-Trace

# 2. 后端环境
cd server
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
copy .env.development.example .env
# 编辑 .env，填入 API Keys

# 3. 前端构建（生产模式）
cd ..\client
npm install
npm run build  # 生成 dist/ 目录

# 4. 启动后端（生产模式）
cd ..\server
set APP_PROFILE=release
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000

# 5. 启动桌面壳（可选）
cd ..\desktop
python main.py
```
