# Sky-Trace 工程骨架说明

## 1. 文档目的
本文件用于说明 Sky-Trace 的目标工程骨架、模块职责、关键文件作用与最小联调路径，作为开发初期的统一参考。

## 2. 分支现状说明
当前你在 docs 分支。该分支目前主要包含文档，不包含 server、client、desktop 的代码目录。

因此本说明描述的是“目标骨架”（已定义的代码结构方案），用于后续与 develop 分支代码同步时对齐。

## 3. 目标工程目录结构
建议采用如下目录布局：

```text
Sky-Trace/
  client/                    # Vue3 前端
    src/
      components/            # 航班列表、筛选面板等组件
      views/                 # 页面级视图（地图主页面）
      services/              # HTTP/WS 请求封装
      stores/                # Pinia 状态管理
      router/                # 前端路由
      types/                 # TypeScript 类型定义
  server/                    # FastAPI 后端
    app/
      api/
        routes/              # REST 接口
        ws/                  # WebSocket 接口
      core/                  # 配置与日志
      models/                # Pydantic 数据模型
      services/              # 数据采集/存储/处理服务
      state.py               # 进程级共享状态
    tests/                   # 基础测试
  desktop/                   # PySide6 桌面壳
    app/
      window.py              # 主窗口与 WebEngine 加载
      bridge.py              # Qt WebChannel 桥接
      backend_manager.py     # 后端进程启动/停止管理
    main.py                  # 桌面入口
  scripts/                   # 开发辅助脚本
  docs/                      # 项目文档
```

## 4. 三层模块职责

### 4.1 server（后端）
职责：
- 提供 REST API（航班列表、详情、轨迹、健康检查）
- 提供 WebSocket 实时推送
- 管理采集流程（先 mock，后接真实数据）
- 维护统一数据模型与字段口径

关键能力：
- `GET /api/v1/health`
- `GET /api/v1/flights`
- `GET /api/v1/flights/{flight_id}`
- `GET /api/v1/flights/{flight_id}/track`
- `WS /api/v1/ws/flights`

### 4.2 client（前端）
职责：
- 显示地图、航班点位、轨迹与航班列表
- 管理筛选、搜索、详情交互
- 调用 HTTP 接口并消费 WS 推送
- 维护页面状态（Pinia）与类型约束（TypeScript）

关键能力：
- 首屏加载航班快照
- WebSocket 实时刷新
- 列表面板与地图联动

### 4.3 desktop（Qt 桌面壳）
职责：
- 启动桌面窗口并加载前端页面
- 启动/停止后端进程
- 通过 WebChannel 提供本地能力桥接（后续扩展）

关键能力：
- 生产模式加载 client/dist
- 开发模式回退到 http://localhost:5173

## 5. 关键代码文件说明（目标骨架）

### 5.1 后端关键文件
- `server/app/main.py`：服务入口、生命周期、路由挂载、CORS 配置
- `server/app/api/routes/flights.py`：航班相关 REST 接口
- `server/app/api/routes/health.py`：健康检查接口
- `server/app/api/ws/flights_ws.py`：航班实时 WebSocket 推送
- `server/app/models/schemas.py`：FlightBrief/FlightDetail/TrackPoint 等模型
- `server/app/services/mock_collector.py`：mock 数据生成器
- `server/app/services/flight_store.py`：内存快照与轨迹缓存
- `server/app/core/config.py`：环境变量读取

### 5.2 前端关键文件
- `client/src/main.ts`：前端应用入口
- `client/src/router/index.ts`：路由注册
- `client/src/views/MapView.vue`：地图主视图
- `client/src/components/FlightListPanel.vue`：航班列表面板
- `client/src/services/api.ts`：HTTP 请求封装
- `client/src/services/ws.ts`：WebSocket 封装
- `client/src/stores/flight.ts`：航班状态管理
- `client/src/types/flight.ts`：航班类型定义

### 5.3 桌面端关键文件
- `desktop/main.py`：Qt 程序入口
- `desktop/app/window.py`：QWebEngineView 加载逻辑
- `desktop/app/backend_manager.py`：后端进程管理
- `desktop/app/bridge.py`：Qt 与前端桥接对象

## 6. 最小联调流程（P0）
按以下顺序可最快打通可演示链路：

1. 启动后端
- 确认 `/api/v1/health` 可访问
- 确认 `/api/v1/flights` 有返回

2. 启动前端
- 页面能显示航班列表数量
- WebSocket 接收到 snapshot 消息并刷新

3. 启动 desktop
- 窗口成功加载前端页面
- 关闭桌面窗口时后端进程可正常停止

## 7. 代码注释约定（建议）
为了让骨架更适合协作，建议保留三类注释：

- TODO：必须实现的功能点（例如筛选参数、重连策略）
- NOTE：实现约束或设计原因（例如字段兼容性）
- FIXME：已知缺陷与临时处理（例如开发阶段硬编码）

## 8. 从 docs 分支到代码落地的建议路径

### 路径 A（推荐）
- 先切换到 develop 分支
- 拉取包含骨架代码的最新提交
- 再将 docs 分支文档通过 merge 或 cherry-pick 合入

### 路径 B
- 保持在 docs 分支
- 按本文件目录手工创建骨架
- 后续再与 develop 对齐并处理冲突

## 9. 骨架完成判定标准
满足以下条件即可视为“骨架完成”：

- 三层目录（client/server/desktop）齐全
- 后端健康接口与航班列表接口可访问
- 前端页面可启动并展示基础数据
- desktop 可加载前端并管理后端进程
- 文档与代码目录一致
