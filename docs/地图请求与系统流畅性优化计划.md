# 地图请求与系统流畅性优化计划

## 1. 背景与问题现象

在网络较慢且缩放级别较大时，地图左右拖动会出现：

- 底图只显示大色块，细节纹理延迟明显
- 地名标注（symbol + glyph）加载不稳定
- 请求失败后可见区域恢复慢，体感卡顿

同时，系统内其他 API（航班详情、回放、统计等）在弱网下也存在超时敏感、重复请求、无复用的问题，整体流畅度不足。

## 2. 现状代码勘察结论

### 2.1 前端地图请求链路

- 地图页与回放页均使用 `MapLibre GL`，并通过 `transformRequest` 改写到本地代理前缀：
  - `/maptiler-proxy/*`
  - `/stadia-proxy/*`
- `setStyle(..., { diff: false })` 会触发全量样式重载，导致 style/glyph/sprite 请求高峰。
- 当前未对地图请求做统一降级策略（重试、请求优先级、缓存命中可视反馈）。

### 2.2 后端地图代理

`server/app/api/routes/map_proxy.py` 当前实现：

- 具备基础转发、超时控制、简单 `Cache-Control`
- 缺少关键能力：
  - 条件请求透传（`If-None-Match` / `If-Modified-Since`）
  - 关键头透传（`ETag` / `Last-Modified` / `Expires` / `Vary`）
  - 失败兜底（stale-if-error）
  - 弱网短时重试（尤其 502/503/504）
  - 进程内热点资源微缓存（style/sprite/glyph）

### 2.3 其他 API

- `client/src/services/api.ts` 仅有固定 10s 超时，缺少：
  - GET 请求去重（同参数并发复用）
  - 短 TTL 缓存（读多写少接口）
  - 弱网重试策略（幂等 GET）

## 3. 优化目标（可验收）

### 3.1 地图相关

- 在 200-800ms RTT、5%-10% 丢包模拟条件下：
  - 地图平移后首屏可见区域恢复时间降低 25%+
  - 地名缺失持续时间降低 40%+
  - 地图代理 5xx 比例下降 30%+

### 3.2 系统 API

- 重复 GET 请求数降低 30%+
- 常用只读接口（如机场/AQI/统计）二次打开命中缓存，P95 响应降低 20%+
- 页面无感卡顿减少（主线程阻塞与无效重渲染下降）

## 4. 分阶段改造计划

## 阶段 A：地图代理可靠性与缓存能力（优先级 P0）

### A1. 条件请求与关键响应头透传

- 位置：`server/app/api/routes/map_proxy.py`
- 改造项：
  - 向上游透传请求头：`If-None-Match`、`If-Modified-Since`、`Accept`、`Accept-Encoding`
  - 返回头透传：`ETag`、`Last-Modified`、`Expires`、`Vary`、`Content-Encoding`
  - 支持 304 直返，减少无效下载

### A2. 地图热点资源微缓存（进程内 TTL）

- 目标资源：style JSON / sprite JSON&PNG / glyph PBF（优先）
- 行为：
  - 命中直接返回
  - 上游失败时若有可用缓存，返回 stale 内容并附加 `Warning` 头
- 上限：
  - 条目数限制 + 体积限制（防止内存失控）

### A3. 弱网重试与连接池参数优化

- 仅对幂等 GET 开启 1 次短重试
- 调整 `aiohttp.TCPConnector`：并发连接、keepalive、DNS 缓存
- 超时按连接/读取拆分，避免总超时过短导致频繁 504

## 阶段 B：前端地图请求与渲染流畅性（优先级 P0）

### B1. 地图实例参数优化

- 位置：`client/src/views/MapView.vue`、`client/src/views/PlaybackView.vue`
- 改造项：
  - 在弱网场景降低过度请求震荡（禁止不必要过期资源刷新）
  - 提升平移/缩放期间已请求资源复用率（缓存容量参数）

### B2. 样式切换与图层重建节流

- 避免短时重复触发 `setStyle` 与重建链路
- 将可复用图层刷新与必须重建逻辑分离

### B3. 标注可见性兜底

- 为 `glyph`/`symbol` 失败场景增加更稳健的退化策略与日志提示
- 便于快速定位“仅色块无地名”问题来源（网络 / 字体 / 样式）

## 阶段 C：通用 API 请求优化（优先级 P1）

### C1. 统一 GET 去重与短 TTL 缓存

- 位置：`client/src/services/api.ts`
- 改造项：
  - 同 URL+params 并发请求复用同一 Promise
  - 只读接口短 TTL（5s-60s）

### C2. 幂等 GET 弱网重试

- 对 408/429/5xx 采用指数退避重试（上限 1-2 次）
- 保护机制：非幂等接口不重试

### C3. 请求超时策略分级

- 航班详情、轨迹、回放等设置更符合数据规模的超时上限
- 避免统一 10s 导致“大接口误判失败”

## 阶段 D：可视化监测（按需求跳过）

- 按当前需求，本轮不做新增可视化监测页面与前端监测面板。
- 保留必要后端统计字段，不增加额外可视化实现。

## 5. 实施顺序（本轮）

1. 完成阶段 A（A1+A2+A3）
2. 完成阶段 B 的基础项（B1）
3. 完成阶段 C 的基础项（C1+C2）
4. 本地构建验证 + 回归检查

## 6. 风险与回滚

- 风险：
  - 缓存策略不当可能导致样式更新滞后
  - 重试策略不当可能放大请求洪峰
- 控制：
  - TTL 短周期 + 上限控制
  - 仅幂等 GET 重试且上限严格
- 回滚：
  - 改造集中在 `map_proxy.py` 与 `api.ts`，可按文件级快速回退

## 7. 验收清单

- 弱网模拟下地图平移后，地名恢复明显快于改造前
- 代理统计可看到缓存命中与重试计数
- 前端重复 API 请求显著减少
- `npm run build` 通过，后端服务可正常启动
