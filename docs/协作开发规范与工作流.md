# Sky-Trace 协作开发规范与工作流

## 1. 目标
本规范用于保证双人协作开发过程稳定可控，减少代码冲突与沟通成本，避免后期集成灾难。

## 2. 角色与责任
### 2.1 项目负责人（后端）
- 负责整体架构与技术路线决策
- 负责 Python 服务、数据采集、接口设计与发布节奏
- 负责契约管理、分支策略执行与 PR 质量把关

### 2.2 前端负责人（UI）
- 负责 Vue3 界面设计、地图渲染、交互体验
- 负责前端状态管理、组件规范与页面性能
- 负责基于契约完成联调与前端错误回归

## 3. 目录边界与文件所有权
为减少冲突，明确模块边界。

- client：前端主责，后端不直接改动业务组件
- server：后端主责，前端不直接改动服务逻辑
- desktop：Qt 壳层以项目负责人主责，前端可协助页面嵌入
- docs：双方都可改，但涉及架构、接口、流程必须经过负责人确认

## 4. 分支模型
采用简化 Git Flow。

- main：稳定分支，只接收通过评审与测试的代码
- develop：集成分支，日常联调与阶段验收
- feature/*：功能分支，例如 feature/realtime-stream
- fix/*：缺陷修复分支，例如 fix/ws-reconnect
- docs/*：文档分支，例如 docs/collab-workflow

## 5. 协作工作流（Workflow）
### 5.1 需求进入
- 每个需求先建 Issue，写清目标、边界、验收标准
- 负责人给需求打标签：backend、frontend、shared、urgent

### 5.2 开发领取
- 每个成员从 develop 拉取最新代码
- 创建个人功能分支：feature/姓名缩写-功能名
- 提交前先本地自测，确保不破坏现有能力

### 5.3 提交规范
采用 Conventional Commits。

- feat: 新功能
- fix: 缺陷修复
- refactor: 重构（不改行为）
- docs: 文档更新
- chore: 构建与脚本调整
- test: 测试相关

示例：
- feat(server): add websocket flight stream endpoint
- fix(client): handle map marker flickering on refresh

### 5.4 PR 规范
- PR 必须关联 Issue
- PR 描述必须包含：变更内容、影响范围、测试方式、截图或日志
- PR 规模建议控制在 300 行以内（不含锁文件）
- 至少 1 人审核通过后才允许合并

### 5.5 合并策略
- 默认使用 Squash Merge，保持 main 历史整洁
- 禁止直接向 main 推送
- 合并前必须先同步最新 develop，解决冲突后再提交

### 5.6 联调节奏
- 每天固定一次 develop 集成（建议晚间）
- 集成后进行Smoke验证：服务启动、地图加载、实时数据推送
- 若冒烟失败，优先回滚当次引入变更并记录根因

## 6. API 契约协作机制
- 后端先行定义接口契约（路径、参数、响应字段）
- 前端按契约开发，不依赖临时字段
- 契约变更流程：提出变更 -> 双方评估 -> 同步更新文档 -> 再开发
- 对破坏性变更必须提前至少一个迭代通知

## 7. 冲突预防规则
- 不跨模块大面积格式化代码
- 避免多人同时改同一文件核心区域
- 前端组件和后端服务尽量按新增文件方式扩展
- 每日开始开发前先拉取并 rebase 最新 develop
- 锁文件（package-lock.json）由前端负责人统一处理

## 8. 代码质量门禁
- 后端：通过基础单元测试与接口自测
- 前端：通过构建检查与关键页面手测
- 共用门禁：无明显报错日志、无阻断性缺陷、文档同步更新

## 9. 发布与版本管理
- 里程碑发布采用语义化版本：v0.1.0、v0.2.0
- 发布前打 Release Checklist
- 每次发布记录：新增功能、修复项、已知问题、回滚方案

## 10. 沟通与会议机制
- 每日简短站会（10 分钟）：昨天完成、今天计划、阻塞项
- 每周一次评审：架构风险、进度偏差、质量问题
- 紧急问题在群内同步后，负责人立即裁决处理路径

## 11. 文档与知识沉淀
- 重要设计决策必须记录在 docs
- 线上故障与联调问题需形成复盘条目
- 新成员接入以文档为准，避免口头传递失真

---
本规范由项目负责人维护。若流程与实际节奏不匹配，应通过 PR 进行修订并在周会上确认生效。
